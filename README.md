# Лабораторна робота №1

- **Розмітка на HTML5**: Не просто накидав тегів, а зробив чітку структуру. Використав `header` для назви, `main` для всього робочого простору і розбив усе на логічні `section`. Тепер форма і таблиця з результатами живуть окремо, як і має бути.

- **Семантичні таблиці**: Таблицю зі списком заявок зібрав за всіма правилами: є `thead` для заголовків і `tbody`, куди за допомогою JS динамічно залітають дані. Кожна колонка на своєму місці, нічого не зсувається.

- **Зв'язок полів**: Щоб усе було чітко, кожному інпуту (`input`, `select`, `textarea`) дав свій `id`, а зверху прикрутив `label` через атрибут `for`. Тепер тицяєш на текст — і курсор сам стає в потрібне поле.

- **Варіант №3 (Реєстр обладнання)**: Реалізував форму для ключової сутності **CheckoutRequests**. У ній рівно **6 полів**: код заліза, ПІБ того, хто бере, дата «з», дата «по», статус заявки та поле для коментаря, де можна розписати деталі.

- **Кнопки без «сюрпризів»**: Чітко розділив типи кнопок. На додавання поставив `type="submit"`, щоб спрацьовувала валідація, а на очищення форми — `type="button"`. Тепер, коли скидаєш дані, сторінка не перевантажується і не бісить.

- **Верстка на Flexbox**: Щоб інтерфейс виглядав нормально, використав флекси. Головний контейнер має `display: flex`, що дозволило мені зручно поставити форму зліва, а таблицю — справа. Все вирівняно по сітці.

- **Стабільна ширина**: Вліпив глобальне правило `box-sizing: border-box`. Тепер мені по барабану на внутрішні відступи та межі — вони не розпирають блоки, і розміри елементів завжди такі, як я задав.

- **JS-логіка та LocalStorage**: Написав скрипт, який робить усю «брудну» роботу: зчитує форму, перевіряє помилки, сортує список і фільтрує за статусом. А щоб дані не злітали після оновлення сторінки, налаштував збереження в **LocalStorage** — це типу моя база даних у браузері на цю лабу.

# Лабораторна робота №2

- **Справжній бекенд**: Переніс усю логіку на Node.js та Express. Тепер це не просто картинка в браузері, а реальний сервер, який приймає запити. Дані тримаю в оперативці (in-memory), щоб усе літало.

- **TypeScript**: Написав проєкт на TS. Це була гарна ідея — тепер код типізований, і я не витрачаю час на пошук тупих помилок з типами.

- **Розподіл по шарах**: Не звалював усе в один файл. Розділив код на контролери (приймають запити), сервіси (думають) та репозиторії (працюють з даними). Тепер усе лежить на своїх місцях.

- **Middleware та логування**: Прикрутив логер, який пише в термінал кожен рух, і централізований обробник помилок. Якщо хтось введе фігню — сервер не впаде, а чітко скаже, що не так.

- **Автотести**: Налаштував Jest та Supertest. Написав тести, які перевіряють, чи сервер правильно створює, віддає та видаляє заявки. Зелені галочки в терміналі гріють душу.

- **Робота з API**: Оскільки фронтенд прибрав, тестую все через термінал за допомогою cURL. Розібрався з кодами статусів (201, 204, 400, 404), тепер API працює як годинник.

- **CORS та статика**: Налаштував доступ так, щоб сервер міг віддавати фронтенд, якби я захотів його повернути. Але поки що тримаю суворий бекенд-стайл.

# Лабораторна робота №3

- **Справжня база даних**: Викинув милиці з оперативною пам'яттю і прикрутив SQLite. Тепер усі заявки, юзери та залізо надійно лежать у локальному файлі `app.db`. Саму базу і важкий `node_modules` у Гіт не пушу — усе чисто і по правилах.

- **Міграції замість хаосу**: Не став ліпити створення таблиць абияк. Написав скрипт-мігратор, який сам читає SQL-файли і по черзі розгортає три пов'язані таблиці: `Users`, `Equipment` та `CheckoutRequests`.

- **Жорсткі зв'язки (Foreign Keys)**: Зв'язав таблиці намертво. Увімкнув `PRAGMA foreign_keys = ON`. Тепер база сама контролює цілісність: видати обладнання неіснуючому студенту не вийде, сервер обробить це і культурно відіб'є запит.

- **Сирий SQL та JOIN'и**: Жодних ORM чи магічних фреймворків. Усі запити пишу ручками. Зробив повноцінний `JOIN`, щоб у списку заявок виводились нормальні імена людей і назви техніки, а не голі айдішники. Плюс додав агрегацію (`COUNT`) для виведення статистики по статусах.

- **Діра в безпеці (фіча, а не баг)**: Як і вимагає методичка, збираю SQL-запити "в лоб" через конкатенацію рядків. Залишив спеціальний ендпойнт із `LIKE`, який спеціально зроблений вразливим до SQL-ін'єкцій. Це свідома підготовка бази для виконання п'ятої лаби.

- **Автоматичний Сідер**: Набридло щоразу забивати дані руками для тестів. Написав скрипт `seed.ts`, який автоматично закидає в базу стартових юзерів, інвентар і тестову заявку. Запустив один раз — і маєш готовий робочий полігон.

- **Розумна обробка помилок**: Якщо хтось намагається вставити дублікат пошти чи інвентарного номера, база не просто крашиться. Централізований `errorHandler` ловить `UNIQUE constraint failed` і красиво віддає клієнту 409 Conflict. Для кривих даних летить 400 Bad Request.
